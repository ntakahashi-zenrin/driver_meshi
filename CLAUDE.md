## 0. このプロジェクトのゴール（今回のスコープ）

**目的：依頼者のPC上（ローカル）に、Appwriteを土台にした「トラックドライバー飯SNS」の
動くサンプルを構築し、企画として成立するかを検証できる状態にすること。**

今回はネット公開やアプリ化はしない。「手元で動かして触って評価する」ことがゴール。
ただし、ここで作るデータ構造・画面は、将来Web公開やスマホアプリ化に進むときの
土台として再利用できるよう、素直で標準的な作りにすること。

このプロジェクトが何のSNSなのか（背景・思想・画面要件）は、企画書
`truck_driver_meshi_kikakusho.md` にすべて書かれている。**実装前に必ず通読すること。**
特にセクション 18〜22（主要3画面）、26〜28（検証ポイントと注意点）が重要。

## 1. すでにあるもの（資産）

- `truck_driver_meshi_kikakusho.md` … 企画書。何を作るかの正本（しょうほん）。
- `driver_meshi_prototype.html` … 動く静的プロトタイプ（HTML1枚）。
  目指す見た目・画面構成・挙動の見本。配色はオレンジ系(#F4661E)・白基調の親しみやすいSNS。
  ※データ保存なし・写真は仮素材。これを「機能する状態」にするのが今回の作業。
- `DATA_MODEL.md` … サンプルで使うデータ構造の定義。Appwriteのセットアップはこれに沿う。

## 2. 技術構成（今回のローカル構成）

- バックエンド：**Appwrite（セルフホスト / Docker）**。安定版 1.9.0 系を想定（2026年時点）。
  認証・データベース・ストレージ・リアルタイムを一式提供するOSSのBaaS。
- フロントエンド：**プロトタイプのHTML/JS を土台にした簡易Webフロント**。
  まずはVite + 素のJS（または軽量に React）で、Appwriteの Web SDK からデータを読み書きする。
  ※凝ったフレームワーク構成にしない。検証用に最小限で動くことを優先。
- 依頼者環境：Docker Desktop 導入済み。ローカルでの実行は可能と確認済み。

> 重要：この技術構成は「ローカル検証用の第一候補」。将来、開発者に引き継ぐ際は
> AppwriteをSupabase等に差し替える判断もあり得る（土台＝厨房は差し替え可能で、
> 本質は DATA_MODEL.md と画面定義にある）。今回はAppwriteで素直に組むこと。

## 3. Appwrite ローカル構築の進め方（2026年時点の手順。着手時に最新を確認すること）

Appwriteは Docker で動く。公式の推奨は、ワンコマンドのDockerインストーラー。

1. 作業用フォルダを作り、以下を実行（Docker Desktop が起動している前提）：
   ```
   docker run -it --rm \
     --volume /var/run/docker.sock:/var/run/docker.sock \
     --volume "$(pwd)"/appwrite:/usr/src/code/appwrite:rw \
     --entrypoint="install" \
     appwrite/appwrite:1.9.0
   ```
   ※バージョン番号は着手時の最新安定版を https://appwrite.io/docs/advanced/self-hosting/installation
   　で確認し、固定（pin）すること。`latest` は使わない。
2. 対話式セットアップに従う。`_APP_OPENSSL_KEY_V1` と `_APP_EXECUTOR_SECRET` は
   推測されない固有の値にする。これらの秘密値は `.gitignore` 対象の `.env` 内に置き、
   絶対にGitにコミットしない。
3. 完了後、ブラウザで http://localhost を開くとAppwriteコンソールが表示される。
   ※非Linux（Mac/Windows）では起動完了まで数分かかることがある（正常な挙動）。
4. コンソールで管理者アカウントを作り、プロジェクトを1つ作成する。
5. 以降、DATA_MODEL.md に沿ってデータベース（コレクション）とストレージを作る。

注意：
- メモリは最低2GB、快適には4GB以上を見ておく（Appwriteは複数コンテナで動くため）。
- インストーラーは20080番ポートでセットアップ用の画面を使う場合がある。
- 破壊的コマンド（`docker compose down -v` は全データ削除）は、実行前に必ず人間に確認する。

## 4. 作る機能（MVP。企画書セクション27「優先度高」に対応）

順番に、各段階の終わりで「ブラウザで動く状態」にして人間が確認できるようにすること。

- 段階1：Appwrite起動 → コンソールでプロジェクト作成 → DATA_MODEL.mdに沿ってDB/ストレージ構築
- 段階2：簡易Webフロントから「ユーザー登録/ログイン」「飯投稿の作成」「飯タイムライン表示」
- 段階3：ドライバープロフィール表示／フォロー・フォロー解除
- 段階4：フォロー中タイムライン（飯投稿＋ギャラリー更新＋バッジ獲得の混在フィード）、いいね

MVPで作らないもの（企画書セクション27「優先度低」）：
ポイント交換、限定テーマ、ランキング、店舗送客、広告、企業タイアップ。
プロフィールカラー・バッジ付与ロジック・コメントは「優先度中」。MVP安定後に着手。

## 5. 守ってほしい企画上の核（外すと別物になる）

- **飯が入口、ドライバーが主役のバランス**（企画書セクション28.4）。投稿カードには
  必ず投稿者のドライバー情報（ニックネーム・車種・走行エリア・タグ）を見せる。
- **未フォローとフォロー中で表示を変える**（セクション10）。未フォロー＝飯投稿中心、
  フォロー中＝飯＋ギャラリー＋バッジ等も流す。
- **入力は選択式・タグ式中心**（セクション14・28.3）。自由入力を増やしすぎない。
- **業務システムっぽくしない**（セクション28.2）。親しみやすいSNSの見た目を保つ。

## 6. 進め方のルール（重要）

1. 一気に全部作らない。段階ごとに動く状態にして人間が確認 → 次へ。
2. 専門用語は避け、平易な言葉で報告する。依頼者は元HTMLコーダーで企画責任者だが、
   現役エンジニアではない。コマンドを実行させるときは何のためのコマンドか一言添える。
3. **破壊的・不可逆な操作（データ削除、課金が発生する設定、外部公開、認証情報の入力）は、
   勝手に進めず必ず人間に確認する。**
4. 秘密情報（Appwriteのキー、APIキー、パスワード）はコードに直書きせず `.env` で管理し、
   `.gitignore` に必ず含める。
5. エラーは普通の工程。出たら原因と対処を分かりやすく説明する。`appwrite` のトラブルは
   公式ドキュメントを参照する。
6. 迷ったら企画書と DATA_MODEL.md に立ち返る。

## 7. 最初のタスク

1. このファイル、`truck_driver_meshi_kikakusho.md`、`DATA_MODEL.md` を読む。
2. `driver_meshi_prototype.html` をブラウザで開いて、目指す見た目と挙動を把握する。
3. 依頼者にDocker Desktopが起動しているか確認し、段階1（Appwrite起動）から着手する。
4. 各段階の終わりに、動作確認の方法（どのURLを開けば何が見えるか）を依頼者に伝える。
